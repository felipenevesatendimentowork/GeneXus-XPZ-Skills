# WorkWithForWeb — Responsibilities and Quality Checklist

Satellite of `xpz-builder/SKILL.md` for the `WorkWithForWeb` object type (`type="78cecefe-be7d-4980-86ce-8d6e91fba04b"`, `Object/@name` starts with `WorkWithWeb`). **Load this file end-to-end before generating, editing, or packaging a `WorkWithForWeb`**, in addition to the main `SKILL.md`.

## Responsibilities

### Structural form detection (Form A vs Form B)

WorkWithForWeb has two distinct XML structural forms depending on context. The 9-WW gate (`Test-GeneXusWorkWithWebApply.ps1`, invoked from WORKFLOW) handles detection automatically, but the agent must be aware of both forms when reasoning about a WorkWithForWeb:

- **Form A (packaging form)**: Part `babfa2b2-19a0-4ef1-b5f4-81b7c7be79dc` with `<Source><Properties>` containing `<Property><Name>Apply</Name>` and `<Property><Name>Transaction</Name>`. Used in `import_file.xml` packages and in XMLs generated from sanitized templates.
- **Form B (acervo form)**: Part `a51ced48-7bee-0001-ab12-04e9e32123d1` with `<Data Pattern="78cecefe-be7d-4980-86ce-8d6e91fba04b">` whose CDATA contains `<transaction transaction="<guid>-<TransactionName>" />`. Used in `ObjetosDaKbEmXml/WorkWithForWeb/*.xml` (materialized acervo).

In Form B, `Apply` is implicitly `True` by construction — a WorkWithForWeb materialized in the acervo was generated by the pattern; the `<Data Pattern="...">` element confirms the pattern is bound.

### CDATA editing rules

When editing the CDATA payload of a WorkWithForWeb (typically Form B), load [01j-workwithweb-cdata-padroes.md](../../01j-workwithweb-cdata-padroes.md) first. The internal XML has at minimum two distinct `<actions>` scopes and often more when Grid tabs are present — empirically, 2/3 of objects in production KBs have `<actions>` in both `<selection>` (list-level) and `<tab>` (detail/grid-level).

- Do not use broad text substitution over repeated tags such as `<actions>`; a pattern-only regex will match across all levels: `<selection>/<actions>` (list actions), `<tab type="Tabular">/<actions>` (per-record actions), and each `<tab type="Grid">/<actions>` (child grid actions).
- Anchor any textual insertion at the intended scope using the parent block's unique structural identifier: `<selection>` for list-level actions; `<tab code="General">` for the main Tabular detail tab; `<tab code="X">` for a specific Grid tab — the `code` attribute is the stable KB-side identifier generated by the pattern and does not vary by locale.
- Confirm any new action appears exactly once in the intended scope; duplicates or ambiguous action scope block packaging.
